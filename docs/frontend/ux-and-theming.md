## UX 规范与主题设计

本节定义 Beehive Web 前端（IM 客户端 + Admin 后台）的基础 UX 规范和主题设计原则，指导日后组件与页面的实现。

---

### 1. 通用 UX 原则

- **一致性**：相同的操作在不同页面保持一致的交互方式与视觉反馈（按钮颜色、文案、位置等）。
- **即时反馈**：所有关键操作（发送消息、保存配置、封禁用户等）都应有明确的 loading 与成功/失败提示。
- **可预期性**：避免「点了不知道发生了什么」，关键操作前给出清晰提示，敏感操作要求二次确认。
- **错误可恢复**：对于网络波动或服务异常场景，给出重试按钮或回退选项，而不是直接失败无解。

---

### 2. IM 客户端 UX 规范

#### 2.1 消息发送与展示

- 消息输入框：
  - Enter：发送消息；
  - Shift + Enter：换行；
  - 输入框聚焦时应清晰，禁用状态时明显区分。
- 发送状态：
  - 发送中：消息气泡淡化并显示「发送中」或进度标记；
  - 发送失败：以红色小图标或文字提示，并提供「点击重试」；
  - 发送成功：去除状态标记。
- 时间显示：
  - 按会话显示消息时间线，时间标签在相邻消息间合理合并（例如超过 5 分钟间隔时显示）。

#### 2.2 会话与未读

- 未读数：
  - 会话列表中左侧展示未读数徽标；
  - 若未读数大于 99，则显示为 `99+`。
- 未读清零规则：
  - 用户切换到某个会话并滚动到底部视为已读，清除该会话的未读计数；
  - 对于系统消息或重要通知，可考虑额外标记。

#### 2.3 键盘与快捷操作

- 支持使用键盘在会话列表中上下切换（可选）；
- 常用操作（如打开设置、搜索会话）支持快捷键（如 `Ctrl+K` 打开全局搜索面板）。

---

### 3. Admin 后台 UX 规范

#### 3.1 列表与筛选

- 列表页均应配备基础筛选区，放在表格上方，支持：
  - 回车触发搜索；
  - 清空筛选条件的快捷操作。
- 表格：
  - 支持列排序（如按创建时间、活跃度排序）；
  - 当数据加载中时显示 skeleton 或 loading 状态；
  - 分页控件固定在下方，显示总数与当前页信息。

#### 3.2 表单与校验

- 表单提交前必须进行前端校验，给出明确的错误提示：
  - 必填项缺失；
  - 格式错误（邮箱、URL、数值范围）。
- 提交成功/失败后，使用全局通知或局部提示明确告知运维人员结果。

#### 3.3 敏感操作

- 对封禁用户、强制下线、修改限流/配置等敏感操作，统一采用：
  - 高亮的警告色按钮（如红色）；
  - 弹出确认对话框，描述后果，并要求用户再次确认。

---

### 4. 主题设计（亮/暗模式）

#### 4.1 基础色板（示意）

- 主色（Primary）：用于主要操作按钮、高亮元素；
- 成功（Success）：用于成功提示；
- 警告（Warning）：用于风险提示；
- 错误（Error）：用于错误提示；
- 中性色（Neutral）：用于背景、边框、分隔线。

Naive UI 与 Element Plus 均支持通过主题配置自定义这些颜色，具体数值可在实现时根据品牌视觉微调。

#### 4.2 暗色模式（IM 客户端优先）

- IM 客户端：
  - 默认支持亮/暗两种主题，用户可在设置中切换；
  - 消息气泡在暗色背景下依然保持足够对比度；
  - 链接、强调文本颜色在暗色模式下需重新校准，避免过亮或过暗。
- Admin 后台：
  - 初期可仅提供亮色主题；
  - 保留后续引入暗色主题的空间。

#### 4.3 主题切换实现建议

- 使用前端全局状态存储当前主题（light/dark），并同步到 localStorage；
- 对于 Naive UI，可以利用其内置的主题切换能力；
- 对于 Element Plus，可以通过 CSS 变量或官方主题机制实现。

---

### 5. 响应式与适配

- IM 客户端：
  - 重点适配桌面浏览器；
  - 在窄屏下（如平板）可将三列布局简化为两列或单列（先显示会话列表，点击进入聊天界面）。
- Admin 后台：
  - 优先考虑桌面端使用，移动端可只保证基本可用。
