## 前端技术栈与组件库选型

### 1. 总体技术栈

- **框架**：Vue 3（组合式 API，为长期演进做准备）。
- **构建工具**：Vite。
- **语言**：TypeScript（推荐），也可在早期阶段允许部分模块使用 JS，逐步迁移。
- **路由**：`vue-router`。
- **状态管理**：`pinia`。
- **HTTP 客户端**：`axios`，统一封装请求/响应拦截器（注入 token、处理错误）。
- **国际化（可选）**：`vue-i18n`。

前端将拆分为两个应用（可放在同一 mono-repo 中）：

- `frontend/apps/client`：IM Web 客户端；
- `frontend/apps/admin`：Admin 管理控制台；

两者共享通用工具与类型（如 WebSocket 消息类型定义、用户/会话 DTO），可放在 `frontend/packages/shared` 或类似目录。

### 2. UI 组件库评选与结论

#### 2.1 评选维度

- Vue 3 支持度与生态成熟度。
- 与场景的匹配度：
  - IM 客户端：聊天窗口、会话列表、弹出层、主题切换等。
  - Admin 后台：表格、表单、复杂筛选、弹窗/抽屉等。
- 主题定制与暗色模式支持。
- 文档质量、社区活跃度和维护情况。

#### 2.2 候选库分析

- **Naive UI**
  - 优点：
    - 为 Vue 3 原生设计，API 风格现代；
    - 主题系统强大，支持亮/暗主题切换与多主题运行时切换；
    - 组件风格简洁，适合 IM 客户端一类偏「产品感」的页面；
    - 通过 Card/List/Scroll 组合和自定义样式，可以快速搭建聊天列表和消息窗。
  - 缺点：
    - 相较 Element Plus，其在「中后台表格/表单生态」上略少一些示例与社区沉淀，但功能上已基本满足需求。

- **Element Plus**
  - 优点：
    - Vue 3 官方推荐之一，延续 Element UI 的成熟生态；
    - 表格、表单、布局、对话框等中后台组件非常完备；
    - 文档详尽，社区和周边生态丰富。
  - 缺点：
    - 默认视觉风格偏传统中后台，直接用于 IM 聊天客户端需要更多自定义 UI。

- **Ant Design Vue**
  - 优点：
    - 源自 Ant Design 设计体系，表单/表格/列表在中后台场景很强；
    - 国际化和布局体系完善。
  - 缺点：
    - 与 Vue 3 的绑定和生态略逊于 Element Plus；
    - 默认风格也更偏中后台，对 IM 产品 UI 需要更多定制。

#### 2.3 选型结论

- **IM Web 客户端（client）**
  - UI 库：**Naive UI**。
  - 理由：
    - 视觉风格现代，方便做聊天类应用的整体观感；
    - 主题系统适合做暗色模式、多主题；
    - 组件数量足以覆盖聊天应用需求，缺少的 Chat 专用组件可以通过组合实现。

- **Admin 管理后台（admin）**
  - UI 库：**Element Plus**。
  - 理由：
    - 中后台常用组件（表格、表单、筛选器、弹窗）齐备；
    - 文档/示例丰富，上手成本低；
    - 对管理类场景（用户/会话/消息查询、配置管理）非常匹配。

### 3. 其他关键前端模块

- **WebSocket 封装**
  - 使用浏览器原生 WebSocket + 自定义 Hook（例如 `useWebSocket`），封装：
    - 自动重连策略；
    - 心跳包发送与超时检测；
    - 将底层帧映射为统一的 JSON Envelope（`type/tid/payload/error`）；
    - 回调/事件总线，将不同类型的消息派发到对应模块。
  - 可选：使用 `reconnecting-websocket` 或类似库简化重连逻辑。

- **表单校验**
  - 优先使用 UI 库内置的校验能力（Naive UI / Element Plus 自身的表单规则）。
  - 对于复杂场景，可引入 `yup`/`zod` 进行 schema 校验，并与 UI 表单集成。

- **图表库（Admin 使用，可选）**
  - 推荐：ECharts（通过 `vue-echarts`）或 AntV G2Plot；
  - 用于仪表盘中的时间序列指标和分布图展示。

- **工具与代码组织**
  - WebSocket 协议类型定义（消息 `type`、payload 结构）抽离到共享包，以保证前后端协议对齐；
  - HTTP API client（针对 AdminAPIService）统一封装到 `api` 模块，避免在组件内散落请求细节。
